## 思政融入（在课设任务书每阶段标注）

### 思政融入点（对应物资采购库存管理的课设场景）

| 课程设计阶段 | 思政融入点 | 融入时机 | 正例 | 反例 |
|-------------|-----------|---------|------|------|
| **需求分析** | 数据安全意识：业务数据是企业的核心资产，调研时需遵守数据保密 | 导入/讨论 | "在调研采购流程时，提醒学生注意供应商信息、合同金额等数据的敏感性，引导学生讨论数据保密的重要性" | "直接让学生下载真实企业的采购数据库练习" |
| **概念结构设计** | 系统思维：站在全局视角设计企业信息系统的数据流转 | 讲解 | "用 E-R 图展示采购-库存-财务的数据流转，强调'局部服从整体'的系统设计思想" | "只让学生做一个表的概念设计，不看全局" |
| **逻辑结构设计** | 工程严谨：规范化理论要求精确的数据依赖分析，是对工程质量的基本要求 | 讲解/作业 | "指出模式分解不彻底会导致数据冗余和更新异常，类比工程领域的'偷工减料'——看似省事实则埋雷" | "直接给完整的关系模式让学生照着建表，跳过规范化分析" |
| **数据库实现** | 工匠精神：SQL 代码规范、正确注释是对自己和他人的负责 | 实操/检查 | "要求每条 SQL 写功能注释、规范命名，教师随机抽查，强调代码'可读性就是可维护性'" | "不要求注释，只要运行对就行" |
| **功能调试** | 数据完整性是底线：触发器、约束是保证数据正确的最后防线 | 检查 | "故意在数据表上不设外键约束，让学生亲眼看到数据不一致的后果，再讨论完整性对企业的意义" | "直接告诉学生'这里要加外键'" |
| **团队协作** | 信息共享与责任意识：数据库设计不是一个人的事 | 答辩 | "要求团队展示各自负责的模块如何衔接，并回答'如果你的部分出了问题，对全团队有什么影响'" | "只收个人报告，不做小组答辩" |

### 融入方式说明

| 方式 | 说明 | 适用阶段 |
|------|------|---------|
| **导入** | 课程设计启动时，用真实案例/新闻引出思政主题 | 需求分析、团队协作 |
| **讲解** | 老师讲解技术点时自然带出思政内涵 | 概念设计、逻辑设计 |
| **讨论** | 设计课堂/线上讨论环节 | 需求分析 |
| **实操** | 在动手实践中融入思政 | 数据库实现、功能调试 |
| **检查** | 评分点评时指出思政相关的好/不足 | 全过程 |

### 融入原则

1. **每阶段不多于 2 条**，每条 1-2 句话
2. **不单独设"思政章节"** — 融入在教学内容和考核标准中
3. **不写虚词** — 如"培养学生的……」"这类空话不算，要有具体场景
4. **正反对比** — 正例告诉老师"这就算融入了"，反例告诉老师"这个就过了"

## 附录：课设任务书（初版）结构预览

生成的课程设计任务书（初版）将包含以下章节，结构来源于课设大纲：

```
《数据库原理与应用（SQL-Server）实践》课程设计任务书

一、课程设计目标
    （从课设大纲 §二+§三 提炼）
    目标 1：理解数据库理论…掌握数据库系统设计的完整方法
    目标 2：掌握 SQL 语言实现…提升实践与协作能力

二、设计任务
    任务一：物资采购和库存管理数据库设计（单元1）
        · 需求分析阶段
        · 概念结构设计阶段（E-R 图）
        · 逻辑结构设计阶段（关系模式、规范化）
    任务二：物资采购和库存管理数据库实现（单元2）
        · 数据库开发（建库/建表/视图/触发器）
        · 功能模块设计实现（存储过程/T-SQL）
        · 调试与维护

三、进度安排
    （从课设大纲 §六 学时分配表衍生）
    第 1-2 天：需求分析 + 概念结构
    第 3-4 天：逻辑结构 + 数据库实现
    第 5 天：功能调试 + 撰写报告
    第 6 天：答辩准备 + 答辩

四、考核标准
    平时考核 20% + 设计报告 60% + 答辩考核 20%
    （从课设大纲 §八 细化）

五、思政融合要求
    （融入在每个任务阶段的教学要求中，详见思政融入点表）

六、交付物清单
    （详见《作业要求》章节）

七、参考资料
    教材 + MOOC + AI 工具 + 任课教师提供
```
